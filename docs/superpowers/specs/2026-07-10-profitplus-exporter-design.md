# ProfitPlus Exporter — Design Spec

**Date:** 2026-07-10  
**Status:** Approved  
**Stack:** Next.js App Router · TypeScript · Bun · Drizzle ORM · better-sqlite3 · mssql · Tailwind · jose · argon2

---

## 1. Purpose

A lightweight internal web application that queries a SQL Server ERP (Profit Plus) and exports custom reports to CSV. The ERP's built-in reports are rigid and unoptimized; this app provides flexible date filtering, column visibility control, and drag-to-reorder before triggering a clean CSV download.

---

## 2. Architecture Overview

Two databases. One app. No external services.

| Concern | Database | Driver / ORM |
|---|---|---|
| User accounts & auth | SQLite (`data/app.db`) | `better-sqlite3` + Drizzle ORM |
| ERP report data | SQL Server (Profit Plus) | `mssql` singleton connection pool |

Sessions are stateless JWTs stored in `httpOnly` cookies. No session table needed.

## 4. Database Layer

### 4.1 SQLite Schema (`lib/db/schema.ts`)

```typescript
users {
  id            integer  PRIMARY KEY AUTOINCREMENT
  email         text     UNIQUE NOT NULL
  name          text     NOT NULL
  password_hash text     NOT NULL          // argon2id
  role          text     NOT NULL          // 'user' | 'admin'
  created_at    integer  NOT NULL          // unix timestamp ms
}
```

No sessions table. Sessions are encoded in signed JWTs.

Future extensibility: add `token_version integer default 0` to `users` if forced-logout on password reset is needed. Increment it to invalidate all existing JWTs for that user.

### 4.2 MSSQL Pool (`lib/db/mssql.ts`)

- Module-level singleton, lazily initialized on first `getPool()` call
- All route handlers call `getPool()` — never instantiate connections directly
- Prevents connection pool leaks on Next.js hot reload in development
- All config via environment variables (see Section 9)

### 4.3 Report Queries

Supports two source types per report:

| `queryType` | Execution |
|---|---|
| `'view'` | `SELECT * FROM [sourceName] WHERE [dateColumn] BETWEEN @startDate AND @endDate` |
| `'procedure'` | `EXEC [sourceName] @startDate, @endDate` |

`@startDate` and `@endDate` are always bound as `mssql` typed parameters — no string interpolation.

## 5. Auth System

### 5.1 Password Hashing

`argon2id` via the `argon2` package. Chosen over bcrypt due to:
- No 72-byte input truncation
- OWASP recommended (2023+)
- Memory-hard by default (resistant to GPU cracking)

### 5.2 Session Flow

```
Login
  POST /api/auth/login
  → lookup user by email in SQLite
  → argon2.verify(stored_hash, submitted_password)
  → sign JWT { sub: userId, role, name, iat, exp: now+7d }
  → Set-Cookie: session=<token>; HttpOnly; Secure; SameSite=Lax; Path=/

Every protected request
  → middleware.ts reads "session" cookie
  → jose.jwtVerify(token, JWT_SECRET)
  → on failure: redirect to /login
  → on success: forward user payload in x-user-payload header

Admin-only routes
  → route handler reads x-user-payload header
  → verifies role === 'admin'
  → 403 if not

Logout
  POST /api/auth/logout
  → Set-Cookie: session=; Max-Age=0
  → redirect to /login
```

`JWT_SECRET` is a random 32-byte hex string stored in `.env.local`. Never committed.

### 5.3 Admin Password Reset

```
Admin submits: { newPassword, confirmPassword }
POST /api/admin/users/[id]/reset-password
  → verify caller JWT role === 'admin'
  → validate newPassword === confirmPassword
  → argon2.hash(newPassword)
  → UPDATE users SET password_hash = ? WHERE id = ?
  → 200 OK
```

No email, no tokens, no expiry. Admin communicates new password to user out-of-band.

### 5.4 Seed Script (`scripts/seed.ts`)

Run via `bun run seed`. Behavior:
1. Runs Drizzle migrations automatically on a fresh DB
2. Prompts interactively: Email → Name → Password (hidden) → Confirm password
3. Validates email uniqueness
4. Hashes with argon2 and inserts with `role = 'admin'`
5. Prints success confirmation

## 6. Report Registry

### 6.1 Config Types (`lib/reports/registry.ts`)

```typescript
interface ColumnDef {
  key: string             // exact SQL column alias
  label: string           // Spanish display name
  defaultVisible: boolean
  alwaysVisible?: boolean // cannot be toggled off (e.g., document ID)
}

interface ReportConfig {
  id: string              // "ventas" | "compras"
  label: string           // "Libro de Ventas"
  queryType: 'view' | 'procedure'
  sourceName: string      // SQL object name
  dateColumn?: string     // required when queryType === 'view'
  columns: ColumnDef[]
}

export const REPORTS: Record<string, ReportConfig> = {
  ventas: { ...},   // populated in lib/reports/ventas.ts — columns TBD by user
  compras: { ... }, // populated in lib/reports/compras.ts — columns TBD by user
}
// NOTE: column definitions for ventas and compras are intentionally empty stubs.
// The user will supply SQL view/SP names and column keys + labels once they have
// access to the ERP database. The registry shape is final; only the data is pending.
```

Adding a new report: create `lib/reports/<name>.ts`, add entry to `REPORTS`. No changes to API routes or UI components.

### 6.2 Default Date Logic

Computed server-side on every export/preview request when params are absent:

```
Given today = 2026-07-10:
  defaultStart = 2026-06-01  (first day of previous month)
  defaultEnd   = 2026-06-30  (last day of previous month)
```

Uses plain `Date` arithmetic. No date library dependency.

## 7. CSV Export (`lib/csv.ts`)

- **Encoding:** UTF-8 with BOM (`﻿` prepended) — required for Excel on Spanish Windows to auto-detect encoding without the import wizard
- **Quoting:** RFC 4180 — wrap field in `"..."` if it contains `,`, `"`, or `\n`; escape inner `"` as `""`
- **Nulls:** `null` / `undefined` → empty string
- **Headers:** `Content-Type: text/csv; charset=utf-8`, `Content-Disposition: attachment; filename="ventas_2026-06.csv"`
- **Column filtering:** only columns present in the `cols` query param are included, in the order specified

Export route signature:
```
GET /api/reports/[report]/export
  ?startDate=2026-06-01
  &endDate=2026-06-30
  &cols=ruc,razon_social,monto   // ordered list of column keys
```

If `cols` is absent, the export defaults to all columns where `defaultVisible: true`, sorted by `defaultOrder`. This ensures a sensible export even when the client sends no column config.

Preview route (`/api/reports/[report]/preview`) accepts the same `startDate`, `endDate`, and `cols` params and returns JSON with the first 100 matching rows. No CSV generation.

```
┌─────────────────────────────────────────────────────┐
│  ◆ ProfitPlus Exporter              [Ana] [Salir]   │
├──────────┬──────────────────────────────────────────┤
│ Reportes │                                          │
│  Ventas  │   [Report content area]                  │

│ Admin    │   (visible only to role=admin)            │

Fixed-width sidebar. Active route highlighted. Admin nav section is conditionally rendered — only visible when the session JWT carries `role === 'admin'`. Direct URL access to `/admin/users` by a non-admin is blocked server-side: the page component reads the session and redirects to `/reports/ventas` if role check fails (middleware only verifies JWT validity, not role).

### 8.3 Column Manager

- Chip/badge per column: click to toggle visibility
- `alwaysVisible` columns rendered without toggle control, still draggable
- Drag-to-reorder via `@dnd-kit/sortable`
- State is local to the page (React `useState`), resets on navigation

### 8.4 Date Range Picker

- Two `<input type="date">` fields pre-filled with previous-month defaults
- "Aplicar" button triggers preview fetch and updates row count badge
- Client-side validation: start ≤ end

### 8.5 Admin Users Page

```
┌─ Usuarios ──────────────────────────────────────────────┐

- Tailwind CSS only — no component library
- Neutral gray palette, single blue accent
- Target: 1280px desktop width minimum
- No decorative elements — scannable, high contrast

JWT_SECRET=generate-with-openssl-rand-hex-32

## 11. Deployment Target

Windows Server running IIS with:
- URL Rewrite + ARR configured as reverse proxy to `http://localhost:3000`
- Next.js production build (`bun run build`)
- Node.js process managed by NSSM (Non-Sucking Service Manager) as a Windows Service
- `data/app.db` stored outside the app directory for persistence across deploys

Detailed step-by-step deployment in `INSTRUCTIONS.md`.
